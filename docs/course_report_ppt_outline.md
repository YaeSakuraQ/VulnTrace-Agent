## 第 2 页：选题背景与问题定义

- **标题**：为什么要做安全智能体

### 传统渗透测试的三个问题

| 问题 | 具体表现 |
|------|----------|
| 工具割裂 | Nmap、HTTP 探测、漏洞验证、报告生成彼此独立，流程难以串起来 |
| 人工负担重 | 需要人工判断下一步做什么，初学者容易卡在"发现了什么"和"接下来做什么"之间 |
| 缺少闭环 | 扫描结果、审批记录、证据文件、最终报告往往分散，难形成教学演示链路 |

### 本项目要回答的问题

> 能否构建一个面向授权靶场的安全智能体，让 LLM 负责规划，让受控工具负责执行，让审批和审计机制负责边界控制，并最终自动生成结构化报告？

## 第 3 页：项目目标与系统定位

- **标题**：项目目标：不是"自动攻击"，而是"受控验证"

### 目标拆解

- 自动创建并运行授权范围内的测试任务
- 让智能体按状态机逐步推进：发现资产 -> 识别服务 -> 验证风险 -> 生成报告
- 用审批、工具白名单、参数校验和日志审计控制风险
- 让前端页面能完整展示任务运行、思考、审批、证据和 Usage

### 系统边界

| 项目定位 | 说明 |
|----------|------|
| 适用范围 | 本地靶场或课程授权目标 |
| 默认策略 | 默认拒绝公网目标 |
| 执行方式 | 只允许调用预注册工具，不允许模型自由拼接 Shell |
| 结果输出 | 事件日志、证据文件、Markdown/HTML 报告 |

## 第 5 页：技术栈全景

- **标题**：技术选型：前端、后端、Agent 三层协同

| 层次 | 当前技术栈 | 作用 |
|------|------------|------|
| 前端 | Vue 3、Vite、Naive UI、Axios、Vue Router、marked、DOMPurify | 任务工作台、详情页、报告渲染、实时交互 |

### 这一页建议强调

- 后端负责"任务管理与接口编排"
- Agent 层负责"思考和推进"
- 三层共同完成"从任务到报告"的完整闭环

## 第 6 页：系统架构与接口流转

- **标题**：系统架构：页面、API、状态机如何协同

### 建议用一张分层图表达

```text
TaskForm / Task Overview / Task Detail
                │
         REST + WebSocket
                │
 FastAPI(tasks / approvals / reports / learning)
                │
 TaskService + TaskRuntime + ApprovalService
                │
 PentestGraphRunner (LangGraph)
      │            │             │
  Planner      ToolExecutor   ReportAgent
      │            │             │
   LLM/知识库     Kali工具/HTTP原语   Markdown报告/证据
```

### 可点名的接口

- `GET /api/tasks/templates`：拉取演示模板
- `POST /api/tasks`：创建任务，可配合 `auto_run=true`
- `POST /api/tasks/{id}/run|pause|stop`：控制任务
- `GET /api/reports/{id}`：拉取报告
- `GET /api/tasks/api-usage`：拉取 LLM Usage 统计
- `WebSocket /ws/tasks/{id}`：向详情页实时推送事件和任务快照

## 第 7 页：智能体核心原理 - LangGraph 状态编排

- **标题**：任务不是一次性执行，而是状态机循环推进

### 建议展示的主流程

```text
validate_scope
   ↓
choose_action
   ↓
risk_check ──→ human_approval
   ↓               ↓
execute_tool → parse_result → reflect_result
                           ↓
                    decide_continue
                      ↓       ↓
               choose_action  generate_report
```

### 汇报要点

- 每轮都是"观察 -> 决策 -> 执行 -> 反思 -> 再决策"
- 风险高的动作不会直接执行，而是转入审批节点
- 任务结束后自动进入报告生成阶段
- 这比传统脚本式流水线更适合处理不确定目标和多阶段验证

## 第 8 页：混合决策与知识库增强

- **标题**：LLM 不是单独工作，而是和规则、知识库一起决策

### 三层决策机制

| 层级 | 作用 | 当前实现 |
|------|------|----------|
| LLM 规划 | 根据上下文选择下一步工具和参数 | PlannerService 输出结构化决策 |
| 规则兜底 | LLM 不稳定时保证流程还能继续 | 端口映射、阶段策略、启发式工具选择 |
| 知识增强 | 利用漏洞家族和签名帮助定位验证路径 | 43 个 exploit families、119 条 signatures |

### 可重点讲的两个数字

- **31 个注册工具**：模型只能选工具，不能自由发挥到失控
- **43 个漏洞家族 / 119 条签名**：让验证动作更接近"有依据的 exploit 推进"，而不是盲扫

## 第 9 页：安全控制与审计机制

- **标题**：为什么这个系统适合课程演示：它是可控的

### 五层控制体系

| 控制层 | 机制 | 课程汇报中的说法 |
|--------|------|------------------|
| 范围控制 | ScopeGuard 校验目标 | 默认拒绝公网，只允许授权范围 |
| 工具控制 | ToolExecutor 白名单 | 模型只能调用预定义 31 个工具 |
| 参数控制 | Pydantic Schema 校验 | 工具参数不会随意越界 |
| 审批控制 | 高风险动作人工审批 | 把"人类在环"真正放进系统里 |
| 审计控制 | Event + Artifact + Report | 每一步有日志、有证据、有最终报告 |

### 风险容忍度

- `strict`：更保守，适合强调安全边界
- `moderate`：默认教学演示模式
- `relaxed`：适合隔离靶场快速验证

## 第 15 页：项目亮点、价值与当前不足

- **标题**：项目亮点与局限性分析

### 项目亮点

- 把安全智能体做成了可演示、可操作、可审计的完整平台
- 前端不只是展示结果，还展示了任务推进、审批、学习、报告和 Usage
- Agent 不是纯大模型驱动，而是 LLM + 规则 + 知识库 + 工具白名单的混合结构
- 支持模板化任务创建，适合课程汇报和本地靶场复现

### 当前不足

- 主要面向本地靶场，真实复杂网络环境覆盖还不够
- 漏洞验证能力依赖现有工具和知识库规模，泛化能力有限
- 报告与评估目前更偏演示场景，量化评测指标还不够完整

---

## 第 16 页：改进方向与总结

- **标题**：未来工作与课程总结

### 后续改进方向

- 扩展更多服务类型、漏洞家族和验证工具
- 增强知识沉淀机制，让经验候选更自动化地转成知识资产
- 加强任务评测指标，例如成功率、误报率、审批开销、Token 开销
- 增强报告模板和可视化能力，例如自动生成任务回放图和截图证据
- 探索多智能体协作，例如侦察 Agent、验证 Agent、报告 Agent 分工

### 收束总结

> VulnTrace Agent 的核心价值，不是替代人工渗透测试，而是把"受控验证、过程可视、结果可交付"三件事整合成一个适合教学与靶场演示的安全智能体平台。
