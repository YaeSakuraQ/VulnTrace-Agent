# 基于 LangChain 与 DeepSeek V4 Flash 的本地靶场安全智能体技术文档

## 一、项目定位

本项目拟构建一个运行于 Kali Linux 本地环境的安全验证智能体系统。系统以大语言模型（Large Language Model, LLM）为核心决策单元，结合 LangChain 工具调用机制、LangGraph 状态编排机制、FastAPI 后端服务和前端网页交互界面，实现对本地合法靶场的自动化信息收集、漏洞假设生成、低风险验证、攻击路径编排、证据留存和报告生成。

项目场景限定为本地靶场、课程实验环境、授权测试环境。系统默认禁止扫描公网地址，禁止对未授权主机执行测试，运行模型直接拼接任意 Shell 命令。所有具有副作用的工具调用必须经过范围校验、参数校验、超时控制和日志记录

本项目的核心目标不是追求不可控的全自动攻击能力，而是验证大模型在安全测试流程中的规划、推理、工具协调、证据归纳和报告生成能力。

## 二、需求分析

### 2.1 业务需求

第一，系统应支持用户在网页端创建安全测试任务，输入目标靶机地址、目标网段、测试范围、授权说明和任务目标。

第二，系统应支持大语言模型根据目标信息生成分阶段测试计划，包括资产发现、服务识别、漏洞假设、验证方案、证据整理和报告输出。

第三，系统应支持在 Kali Linux 中调用预定义安全工具，但调用过程必须通过后端封装完成，模型只能选择工具和填写结构化参数，不能直接执行任意命令。

第四，系统应支持对本地靶场的分区式推进。每完成一个阶段，系统根据已获得证据更新状态，并决定是否进入下一阶段。

第五，系统应支持人工介入。对于高风险命令、利用验证、凭据使用、文件写入等动作，前端应弹出审批界面，由用户选择批准、拒绝或修改参数。

第六，系统应自动保存完整运行轨迹，包括模型思考摘要、工具调用参数、工具输出、异常信息、关键证据、阶段状态和最终结论。

第七，系统应生成结构化安全测试报告，内容包括目标范围、测试方法、发现结果、影响分析、证据截图或日志、修复建议、测试边界声明。

### 2.2 功能需求

1. 任务管理功能

用户可以创建、暂停、恢复、终止安全测试任务。每个任务应包含唯一任务编号、目标范围、当前阶段、执行状态、开始时间、结束时间和测试说明。

2. 范围控制功能

系统必须维护目标白名单。只允许访问白名单中的 IP 地址、域名和端口。对于公网地址、内网外部地址、广播地址和未授权地址，系统应直接拒绝执行。

3. 智能规划功能

智能体根据用户输入和已有扫描结果生成测试计划。计划应包含阶段目标、所需工具、预期证据、失败处理策略和停止条件。

4. 工具调用功能

后端封装 Kali Linux 中常用安全工具。工具调用采用结构化接口，例如目标地址、端口范围、扫描强度、输出格式、超时时间。模型不得直接控制完整命令行。

5. 结果解析功能

系统应将工具输出转换为结构化结果，例如主机列表、开放端口、服务版本、Web 指纹、疑似漏洞、错误信息和证据片段。

6. 攻击路径编排功能

系统使用图结构记录资产、服务、漏洞假设、验证结果和阶段关系。每个节点保存证据，每条边表示测试推理关系。

7. 人工审批功能

对于可能产生明显副作用的动作，系统进入等待审批状态。用户可以在网页端查看工具名、目标、参数、风险说明和模型理由，然后决定是否执行。

8. 日志审计功能

系统需要保存每次模型调用、工具调用、审批决策、异常恢复和阶段切换记录。日志应支持按任务编号查询。

9. 报告生成功能

系统可生成 Markdown 或 HTML 格式报告。报告应以证据为中心，避免只给结论。每个安全发现都应包含复现范围、风险说明、证据引用和修复建议。

10. 知识库功能

系统可维护本地知识库，用于保存靶场规则、工具说明、常见漏洞检查清单、课程实验约束和历史任务总结。智能体在规划时可检索相关知识，提高稳定性。

## 三、非功能需求

### 3.1 安全性

系统必须默认拒绝公网目标。工具执行层需要实现目标白名单、命令模板、参数校验、超时限制、输出长度限制和危险关键字拦截。所有高风险工具应启用人工审批。

### 3.2 可控性

智能体应采用状态机方式运行。每个阶段只能调用该阶段允许的工具。系统不能让模型在无约束环境下自由执行命令。

### 3.3 可审计性

每一步操作都应有日志。日志应记录输入、输出、时间、执行者、阶段、工具名、参数摘要和审批结果。报告结论必须能回溯到原始证据。

### 3.4 稳定性

工具执行必须设置超时时间。模型调用失败时应支持重试。工具输出过长时应自动摘要并保留原始文件路径。任务中断后应能从最近状态恢复。

### 3.5 可扩展性

新增工具时，只需要新增一个工具封装文件和一个配置项。新增测试阶段时，只需要扩展状态机节点和阶段策略，不应修改核心框架。

### 3.6 成本控制

由于使用远程大模型 API，系统应控制上下文长度。长工具输出不直接全部传入模型，应先做结构化解析和摘要。系统提示词、工具说明和历史摘要应尽量缓存和复用。

## 四、总体架构设计

系统采用前后端分离架构。

前端负责用户交互、任务展示、审批操作、日志查看和报告预览。后端负责模型调用、状态编排、工具封装、范围控制、数据存储和报告生成。Kali Linux 作为工具执行环境。DeepSeek V4 Flash 作为主要推理模型。

整体架构可以分为六层。

第一层是用户界面层。该层提供任务创建页面、智能体对话页面、阶段进度页面、工具输出页面、审批页面和报告页面。

第二层是后端 API 层。该层使用 FastAPI 提供 HTTP 接口和 WebSocket 实时日志推送。

第三层是智能体编排层。该层基于 LangChain 和 LangGraph 实现模型调用、工具选择、状态转移、异常恢复和人工介入。

第四层是工具执行层。该层封装 Kali 工具，统一提供安全的结构化调用接口。

第五层是数据与证据层。该层保存任务状态、工具输出、证据文件、运行日志、知识库文档和报告文件。

第六层是安全控制层。该层贯穿所有模块，负责白名单校验、参数校验、权限控制、审批控制和审计记录。

## 五、核心技术选型

### 5.1 大语言模型

模型选择 DeepSeek V4 Flash。调用方式采用 OpenAI 兼容接口。模型主要负责测试规划、工具选择、结果解释、漏洞假设生成、路径决策和报告撰写。

建议模型配置如下。

模型名称：deepseek-v4-flash
调用方式：OpenAI 兼容 API
温度参数：0.2 至 0.4
最大输出：按任务阶段限制
输出格式：优先使用 JSON 结构化输出
密钥管理：通过 .env 文件和环境变量管理，不写入代码仓库

### 5.2 智能体框架

LangChain 用于定义工具、模型接口和智能体调用。LangGraph 用于构建状态机，保存中间状态，并支持人工审批和任务恢复。

推荐采用单主控智能体加多个工具模块的方案。课程作业阶段不建议过早设计复杂多智能体系统，因为多智能体会增加不可控性、调试成本和上下文消耗。

可设置以下逻辑角色。

规划器：根据目标和证据生成下一步计划。
工具路由器：选择允许的工具和结构化参数。
结果分析器：解析工具输出，更新资产和漏洞假设。
风险控制器：判断动作是否需要人工审批。
报告生成器：将证据整理为技术报告。

### 5.3 后端框架

后端采用 Python + FastAPI。FastAPI 适合构建类型明确的 API 服务，能够自动生成接口文档，并且便于和 Pydantic 数据模型结合。

主要后端模块包括 API 路由、智能体服务、工具服务、任务服务、日志服务、报告服务和数据库访问层。

### 5.4 前端框架

前端可采用 Vue 3 + Vite + Element Plus。该组合适合快速构建课程项目界面，开发成本较低。页面重点不在炫技，而在清楚展示智能体的执行过程和可控边界。

主要页面包括任务管理页、智能体控制台、工具日志页、审批页、知识库页和报告页。

### 5.5 数据存储

课程作业阶段建议使用 SQLite。原因是部署简单、便于提交、便于本地复现。后续如需支持多人协作或大规模任务，可替换为 PostgreSQL。

证据文件和原始工具输出建议保存在本地 artifacts 目录中，数据库中只保存文件路径和摘要。

## 六、运行环境设计

### 6.1 基础环境

操作系统：Kali Linux
后端语言：Python 3.11 或更高版本
前端环境：Node.js LTS
后端框架：FastAPI
智能体框架：LangChain、LangGraph
模型接口：DeepSeek API
数据存储：SQLite
运行方式：本地启动前端和后端服务

### 6.2 网络环境

建议采用虚拟机 Host Only 网络或 NAT 网络。靶机可以使用本地 Docker 靶场、Metasploitable、DVWA、Vulhub 或课程提供的专用靶场。系统默认不允许访问公网目标。

建议网络结构如下。

Kali Linux：运行智能体系统和安全工具。
Target VM：运行靶场服务。
Host Machine：访问前端网页。
网络模式：Host Only 或隔离 NAT。

### 6.3 环境变量

后端应使用 .env 文件保存敏感配置。

DEEPSEEK_API_KEY：DeepSeek API 密钥
DEEPSEEK_MODEL：deepseek-v4-flash
DEEPSEEK_BASE_URL：DeepSeek OpenAI 兼容接口地址
DATABASE_URL：SQLite 数据库路径
ARTIFACT_DIR：证据文件目录
ALLOW_PUBLIC_TARGETS：默认 false
DEFAULT_TIMEOUT：工具默认超时时间

## 七、智能体工作流程

### 7.1 阶段一：任务初始化

用户输入目标范围。系统首先验证目标是否属于白名单。如果目标不合法，任务直接终止。若目标合法，系统创建任务记录，并初始化状态图。

### 7.2 阶段二：资产发现

智能体调用低风险信息收集工具，识别存活主机、开放端口和基础服务。该阶段只允许执行被动或低强度主动探测。

### 7.3 阶段三：服务识别

系统根据开放端口识别服务类型、版本信息和 Web 指纹。工具输出被解析为结构化服务列表。

### 7.4 阶段四：漏洞假设生成

模型基于服务版本、页面指纹、错误信息和知识库内容生成漏洞假设。每个假设必须包含依据、验证方式、风险等级和预期证据。

### 7.5 阶段五：低风险验证

系统优先执行无破坏性验证，例如访问特定页面、检查配置暴露、检测默认页面、验证弱配置迹象。高风险验证动作进入人工审批流程。

### 7.6 阶段六：受控利用验证

仅在本地授权靶场中，且用户明确审批后，系统才允许执行利用验证。执行结果需要保存证据，并更新攻击路径图。课程展示时可以用模拟漏洞或已知靶场漏洞作为演示对象。

### 7.7 阶段七：路径编排

系统将已发现资产、服务、漏洞和证据构建成路径图。模型根据路径图判断下一步最有价值的测试方向。

### 7.8 阶段八：报告生成

系统汇总任务过程，生成 Markdown 或 HTML 报告。报告内容应强调授权范围、测试边界、发现结果、证据链和修复建议。

## 八、工具封装设计

工具层不能向模型暴露任意命令执行能力。推荐每个工具封装为一个 Python 函数，输入参数由 Pydantic Schema 限定，输出统一转换为 JSON。

可规划以下工具类别。

1. 资产发现工具

功能：识别本地靶场中的存活主机和开放端口。
风险等级：低。
审批策略：白名单内可自动执行。

2. 服务识别工具

功能：识别端口服务、协议类型和版本信息。
风险等级：低至中。
审批策略：白名单内可自动执行，扫描强度受限。

3. Web 指纹工具

功能：识别标题、状态码、响应头、常见路径、技术栈。
风险等级：低。
审批策略：白名单内可自动执行。

4. 目录枚举工具

功能：在靶场 Web 服务中检查常见目录和文件。
风险等级：中。
审批策略：限制字典规模和请求速率。

5. 漏洞验证工具

功能：验证本地靶场中的已知漏洞或配置问题。
风险等级：中至高。
审批策略：必须人工审批。

6. 报告工具

功能：生成 Markdown、HTML 或 PDF 报告。
风险等级：低。
审批策略：可自动执行。

工具封装原则如下。

第一，模型只传入结构化参数，可以接触完整 Shell。
第二，后端使用固定命令模板，不允许用户或模型拼接危险参数。
第三，每个工具必须设置超时时间。
第四，每个工具必须限制输出长度。
第五，每次工具调用都必须写入审计日志。
第六，高风险工具必须经过人工审批。
第七，所有目标必须通过白名单校验。

## 九、状态机设计

推荐将智能体运行流程设计为 LangGraph 状态机。

状态字段建议如下。

task_id：任务编号
scope：授权范围
current_stage：当前阶段
hosts：已发现主机
services：已发现服务
hypotheses：漏洞假设列表
evidence：证据列表
actions：历史动作列表
pending_approval：待审批动作
risk_level：当前风险等级
report_path：报告路径
error_count：连续错误次数
stop_reason：任务停止原因

状态节点建议如下。

validate_scope：范围校验
plan_next_step：生成下一步计划
select_tool：选择工具
risk_check：风险检查
human_approval：人工审批
execute_tool：执行工具
parse_result：解析结果
update_path_graph：更新路径图
decide_continue：判断是否继续
generate_report：生成报告
finish：任务结束

状态转移规则如下。

范围不合法，直接进入 finish。
低风险动作，进入 execute_tool。
高风险动作，进入 human_approval。
工具失败，进入 plan_next_step 并附带错误信息。
连续失败超过阈值，进入 generate_report 并说明失败原因。
达到任务目标，进入 generate_report。

## 十、前端页面设计

### 10.1 任务管理页

展示所有测试任务，包括任务名称、目标范围、当前阶段、状态、创建时间和操作按钮。

### 10.2 智能体控制台

左侧显示用户输入和模型阶段性结论。右侧显示当前状态、资产列表、服务列表和漏洞假设。

### 10.3 工具执行日志页

展示每次工具调用的工具名、参数摘要、开始时间、结束时间、退出状态、输出摘要和原始文件路径。

### 10.4 人工审批页

展示待审批动作，包括工具名称、目标、参数、风险等级、模型理由和可能影响。用户可以选择批准、拒绝或修改参数。

### 10.5 攻击路径图页面

使用图结构展示主机、服务、漏洞假设、证据和阶段关系。课程展示时该页面很重要，能体现“路径编排”能力。

### 10.6 报告页面

展示自动生成的安全测试报告，并支持复制、下载 Markdown 或导出 HTML。

## 十一、后端接口设计

建议提供以下接口。

POST /api/tasks
创建新任务。

GET /api/tasks
查询任务列表。

GET /api/tasks/{task_id}
查询任务详情。

POST /api/tasks/{task_id}/run
启动或继续执行任务。

POST /api/tasks/{task_id}/pause
暂停任务。

POST /api/tasks/{task_id}/stop
停止任务。

GET /api/tasks/{task_id}/events
获取任务事件日志。

GET /api/tasks/{task_id}/artifacts
获取证据文件列表。

POST /api/approvals/{approval_id}/approve
批准待执行动作。

POST /api/approvals/{approval_id}/reject
拒绝待执行动作。

POST /api/approvals/{approval_id}/edit
修改参数后批准。

GET /api/reports/{task_id}
获取任务报告。

WS /ws/tasks/{task_id}
实时推送模型输出、阶段变化和工具日志。

## 十三、关键提示词设计

系统提示词应明确以下约束。

你是本地授权靶场安全测试助手。
你只能在用户声明的授权范围内行动。
你不能请求扫描公网目标。
你可以生成或执行未封装的 Shell 命令。
你必须优先选择低风险验证。
你必须在高风险动作前请求人工审批。
你必须基于证据更新结论。
你不能把工具失败解释为漏洞存在。
你必须在报告中区分确认漏洞、疑似风险和未验证假设。

用户任务提示词模板如下。

目标范围：{scope}
靶场说明：{lab_description}
当前阶段：{current_stage}
已有资产：{hosts}
已有服务：{services}
已有证据：{evidence}
允许工具：{allowed_tools}
禁止行为：访问范围外目标、任意命令执行、破坏性操作、未审批利用验证
请输出下一步计划，格式为 JSON。

## 十四、风险控制策略

本项目的风险控制不能只依赖提示词。提示词约束容易被模型忽略，真正有效的控制应写在代码中。

建议实现五类控制。

第一，范围控制。任何工具执行前，必须检查目标是否属于白名单。

第二，工具控制。模型只能调用注册工具，可以直接调用系统 Shell。

第三，参数控制。每个工具参数都必须经过 Schema 校验。

第四，审批控制。高风险工具必须等待用户审批。

第五，审计控制。每个动作必须落日志，报告结论必须能追溯证据。

## 十五、课程展示建议

课程展示可以设计一个小型靶场。靶机部署一个存在弱配置或教学漏洞的 Web 服务。展示流程如下。

第一步，用户在网页端创建任务，输入本地靶机 IP 和授权说明。
第二步，系统验证目标属于白名单。
第三步，智能体生成测试计划。
第四步，系统自动完成资产发现和服务识别。
第五步，智能体提出漏洞假设，并执行低风险验证。
第六步，高风险验证动作进入人工审批。
第七步，用户批准后，系统完成受控验证并保存证据。
第八步，系统生成攻击路径图和最终报告。

展示重点应放在“智能体如何规划、如何调用工具、如何保留证据、如何被约束、如何生成报告”。不要把展示重点放在具体漏洞利用细节上。

## 十六、预期成果

本项目完成后，应形成以下成果。

1. 一个可运行的本地 Web 系统。
2. 一个 Python 后端服务。
3. 一个基于 LangChain 和 LangGraph 的安全智能体。
4. 一组封装后的 Kali 工具接口。
5. 一个任务状态机。
6. 一个工具调用审计系统。
7. 一个本地靶场演示案例。
8. 一份自动生成的安全测试报告。
9. 一份项目设计文档和使用说明。

## 十七、评价指标

可以从以下方面评价系统。

功能完整性：是否完成任务创建、工具调用、审批、报告生成。
安全可控性：是否实现白名单、参数校验、人工审批和审计。
智能体能力：是否能根据证据生成合理下一步计划。
稳定性：工具失败后是否能恢复或停止。
可解释性：每个结论是否有证据支撑。
可扩展性：新增工具和阶段是否方便。
展示效果：前端是否能清楚展示执行过程和攻击路径。

## 十八、实现优先级

第一优先级：任务管理、范围校验、工具封装、模型调用、日志记录。
第二优先级：LangGraph 状态机、人工审批、结果解析、报告生成。
第三优先级：攻击路径图、知识库检索、历史任务复盘、成本优化。
第四优先级：多智能体协作、并发任务、插件市场和复杂靶场联动。

课程作业阶段建议完成前三个优先级。第四优先级可作为展望，不建议一开始实现。

## 十九、项目难点与解决方案

难点一：模型可能产生错误工具调用。

解决方案：后端不执行模型给出的原始命令，只执行结构化工具接口，并对参数进行强校验。

难点二：工具输出过长。

解决方案：保存原始输出到文件，只将摘要和关键字段传给模型。

难点三：智能体容易陷入循环。

解决方案：记录历史动作，限制同类动作重复次数，连续失败后要求模型换策略或停止。

难点四：漏洞结论可能缺乏证据。

解决方案：报告生成阶段强制要求每个结论绑定证据编号。无证据内容只能写为疑似风险。

难点五：高风险动作不可控。

解决方案：设置风险分级。高风险动作必须人工审批，审批前展示目标、参数、理由和影响。

## 二十、总结

本项目可以实现一个面向本地授权靶场的安全智能体原型。系统以 DeepSeek V4 Flash 提供推理能力，以 LangChain 提供工具调用能力，以 LangGraph 提供状态机和人工介入能力，以 FastAPI 提供后端接口，以前端网页提供可视化交互。项目的技术重点应放在安全边界、状态编排、工具封装、证据管理和报告生成。

对于课程作业而言，最有价值的创新点是将“渗透测试流程”转化为“可审计、可恢复、可审批的智能体工作流”。只要范围控制和工具控制做得扎实，该项目既能体现 AI 安全攻防的前沿性，也能满足教学实验的合法性和可控性要求。
